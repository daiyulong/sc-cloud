## 0. 背景与触发

现状（已实现）：Next.js + Prisma 全链路骨架，导航 8 项（工作台/项目/样本接收/实验排期/生信分析/交付/经验库/用户），数据模型 `项目 → 样本(扁平, sampleNo 唯一 + sampleCount) → 实验任务 → 质控/生信`。

三个推动重构的现实：
1. **信息逐步收集**：新建项目时**合同号、样本编号、项目编号都还不知道**，样本名更晚才有；信息在项目过程中逐步补齐，期间部分字段允许暂时空缺。现有"建项目即生成 1 条样本、sampleNo 唯一非空"的模型扛不住。
2. **样本编号 = 批次**：一个样本编号(YP)是**一个样本批次**，含多个样本，每个样本有**样本名**（任务单 BP-G260506372：YP20260505077 → 4 个具名样本 Ctrl_T_1/cKO_T_1/…；经验表：一行一捕获、带样本名）。
3. **多模态录入**：实验员上传实验记录图片，多模态大模型识别**样本名 / 悬液浓度 / 活率 / 结团率**，减少手工录入。这是全新能力，设计稿没有。

## 1. 决策摘要

### 1.1 已锁定决策（老板拍板）

| 决策 | 结论 | 备选与否决理由 |
|------|------|----------------|
| **样本层级** | **两层、单批次**：项目 → **样本批次**(样本编号 YP + 数量 + 到样/接收) → **样本**(样本名 + 逐样本 QC/产出)。一项目当前一个批次。 | ① 三层（送样批次 ⊃ 样本编号 ⊃ 样本）——一次送样含多 YP 暂不存在，过度；② 维持扁平（YP=叶子）——与任务单/经验表真实层级、样本名诉求对不齐。结构上 `SampleBatch.projectId` 天然 1→N，将来要多批次只需放开建批次入口（见 §2.2）。 |
| **多模态方案** | **Vercel AI Gateway 调 Claude 多模态 + 人工确认**：识别值预填表单、实验员核对修正后落库，**绝不自动写质控数据**；原图存证。 | 高置信自动写——质控数值是经验库与交付判断依据，未经人核对落库风险高，否决。 |
| **实现节奏** | **数据模型 + IA 先行**，多模态紧随其后。数据模型本期就**预留**图片/识别字段位。 | 一次做完面太大、不利分步验收；只动 UI 不动模型会返工。 |

### 1.2 采纳的设计理念（来自设计稿 `CLAUDE.md`）

- **工位制 IA，不做待办中转**：导航收敛，全员可见不设墙；每个角色登录直达自己的工位；**砍掉「工作台」聚合页**、铃铛/收件箱/异常独立页/交付独立页。侧栏角标 = 队列深度（不是通知）。
- **队列即工作、动作内联**：要处理的事就在工位列表里，动作按钮在行内；点动作 → **就地抽屉(Drawer)** 或 Dialog 完成（落点规则见 §4），不跳详情页；详情页是次要路径。
- **同事而非权限墙**：角色由登录账号决定、登录后固定；小团队靠"看得见、喊得着"，不为权限牺牲效率。
- **不做运营 Dashboard**：队列 + 角标 + 项目"需关注置顶"已是小团队全部态势感知。仅当将来出现"管经营"的管理角色才做克制的经营看板，默认不做。

### 1.3 复审修订摘要（v2，相对 v1 的实质变化）

> 6 视角对抗式复审（数据模型/IA/多模态/状态机/迁移/一致性）+ 高影响发现验证后的结论。

1. **导航 4+1 → 5+1，保留「生信」工位**（已确认；偏离设计稿的 4 项）：设计稿把生信折进项目，但生信分析员的两条队列（待建生信=实验任务粒度、进行中生信=生信任务粒度）都是任务粒度，项目粒度表达不了，折叠会逼分析员回退到"逐个开详情页做"——正是锁定交互模型禁止的形态。故保留「生信」为全员可见工位（不设权限墙）。仍删工作台、折叠交付、收样独立化，"去冗余导航"目标不变。生信工位另承载产出 API 上报 / 任务单 / 邮件提醒（§6.5）。
2. **产出/QC 维持挂 Sample 叶子**（符合锁定决策#1）：复审一度主张迁到 TaskSample，验证后否决——当前"一样本=一次捕获"，叶子粒度即等价 task_sample 粒度（规格 §9.7「一样本↔一 task_sample」），迁移只增 join 无收益。但补**显式假设 + 退出条件**（§2.3）。
3. **删掉若干伪复杂度**：`batchNo` 用普通 `String? @unique`（同 projectNo，PG 空值天然不冲突），删"部分唯一索引"表述；"写迁移"改为 **db push + reseed**（绿地无真实数据，见 §2.6）；ExperimentTask.sampleId 不留"兼容期"双轨、M1 直接切除（§2.4）。
4. **补全被低估的改动面**：行级可见范围 `lib/auth/role-scope.ts`、经验库查询层 `lib/analytics`、QC/产出录入流、createProjectSchema、测试，全部显式纳入 M1（§8）。
5. **多模态补三处硬缺口**：识别行→占位叶子的**对齐机制**（§6）、私有 Blob 的读取/喂模型方式（§6）、PII 出境隐私条款（§9）。
6. **状态空洞修补**：草稿空批次 + 收样不建叶子会让项目永卡 sample_received；收样强制 count≥1 且建 ≥1 叶子（§5）。

## 2. 数据模型重构（终局）

层级：**Project（委托单）→ SampleBatch（样本批次/YP）→ Sample（样本/样本名）**，外加上机的 `TaskSample` 关联。

### 2.1 Project（项目 = 委托单）
- `projectNo`（项目编号=委托单号）保持可空；草稿用内部 id 定位。
- `contractNo`（合同号）**改为可空**（现 NOT NULL）。
- `createProjectSchema` 同步放开：去掉 `sampleNo`/`sampleCount` 必填、`contractNo` 改可空（现三者均必填，见 `lib/schemas/project.ts`）；`project-form` 对应字段改可选。仅客户单位/联系人 + 服务档次建项目时必填。
- 状态机 `ProjectStatus` 枚举沿用（由子实体聚合驱动，§5）。

### 2.2 SampleBatch（新增，样本批次）
承接旧 `Sample` 的**批次级**字段（编号、数量、到样、接收记录）。

| 字段 | 说明 | 可空 |
|------|------|------|
| `id` | 主键 | — |
| `projectId` | 所属项目（FK，天然 1→N，本期建 1 个） | 否 |
| `batchNo` | **样本编号**（YP…），收样时录入；`String? @unique`（同 projectNo，PG UNIQUE 默认 NULLS DISTINCT：多 NULL 不冲突、有值唯一，**无需部分唯一索引、无需 raw SQL**） | 是 |
| `seq` | 批次序号，默认 1；**加 `@@unique([projectId, seq])`** 为将来多批次留位 | 否 |
| `sampleCount` | 样本数量；收样登记时**必填 ≥1**（见 §5，堵 0 叶子空洞） | 是（建项目时） |
| `species`/`tissueType`/`experimentType` | 批次级共有属性，逐样本可覆盖 | 是 |
| `transportCondition`/`samplingDate`/`expectedArrivalDate` | 运输与到样计划 | 是 |
| `receivedAt`/`receiverId`/`receiveStatus`/`abnormalNote` | 到样接收记录（整批一次接收，**接收时间权威源**） | 是 |
| `status` | 批次状态 | 否 |

**批次态语义（必写，避免接错聚合）**：批次状态是**收样登记结果态、收样完成后不随实验推进改变**；项目/样本生命周期由**叶子**驱动，不要把 `batch.status` 接进项目聚合。收样判定：全部正常→batch `received` + 叶子 `received`；部分样本异常→batch `received` + 对应叶子 `received_abnormal`；整批异常→batch `received_abnormal` + 叶子 `abnormal`。整批异常**仍推进**项目 `sample_received`（沿用现有 isAbnormal 仍 advance 的口径，不污染异常统计）。

### 2.3 Sample（样本，叶子；承接经验表"一行一捕获"）
| 字段 | 说明 | 可空 |
|------|------|------|
| `id` | 主键 | — |
| `batchId` | 所属批次（FK） | 否 |
| `sampleName` | **样本名**；优先由任务单回填、缺失时手工/多模态补（录入路径见 §5） | 是 |
| `species`/`tissueType` | 逐样本属性，默认继承批次 | 是 |
| `status` | 样本状态（待到样/已接收/实验中/已反馈/异常） | 否 |
| `source` | 录入来源 `manual`/`ai_confirmed`（多模态确认标记） | 否（默认 manual） |
| `remark` | 备注（旧 Sample.remark 落叶子） | 是 |

逐样本**质控**（`QcRecord` 改挂叶子 `sampleId`）：`concentration`/`viability`/`aggregationRate`/`qcResult`/`riskLevel`；重构后 QcRecord **收敛为每叶子一条**（`taskId` 仅留作来源引用，不再作多捕获区分键），与产出同处"一行一捕获"粒度。
逐样本**产出指标**（从 `experiment_tasks` 迁来的单值列）：`suspensionType`/`sequencingAmount`/`capturedCells`/`medianGenes`。

**建模假设 + 退出条件（不可从代码反推，必记）**：Sample 叶子 = 一次交付捕获；本期一批次、**不建模重复上机**；补测 = 同一捕获的测序量更新，由可订正的 `recordRunMetrics` 承接（非新捕获行）。产出为 Sample 单值列**是有意为之**（与"一样本=一捕获"一致），非遗漏。**退出条件**：若将来"失败重捕/重复上机作为独立捕获留痕"进入范围，则产出+QC 整体下沉到 `TaskSample` 粒度——`TaskSample`（§2.4）预留即为此退出点。规格 §9.4/§9.7/§6.8 的"task_sample 终局"口径据此改写为"当前 Sample 粒度 = 1:1 下的 task_sample 等价"（列入 §10 同步清单）。

**`projectId` 冗余 FK 取舍**：默认**不在 Sample 加冗余 projectId**，scope/查询经 `{ batch: { project: ... } }` 嵌套过滤（Prisma 支持，少一个需维护的不变量，本期单批次查询量小）。仅当 M1 实测嵌套过滤有性能/可读性问题，才加冗余 projectId 并声明"batch.projectId 为权威、同事务复制、批次不可改挂项目"。

### 2.4 ExperimentTask 与 TaskSample
- `ExperimentTask` = **一次上机/实验事件**（排期、操作人、上机方式、实际日期、result_status/feedback）。
- 引入 **`TaskSample`(taskId, sampleId)** 关联表承接"一次上机多样品"。**M1 直接移除** `ExperimentTask.sampleId` 单值 FK 与 `loadedSampleCount`（绿地无真实数据，不留"兼容期"双轨——双轨=双事实来源、漂移风险）。所有读 `task.sample` 的点（taskInclude、列表搜索、options、analytics、role-scope）改读 `task.taskSamples`。
- **创建流**：实验任务创建改为收**多个 sampleId**（taskSamples 数组），入口从 `/samples/[id]/experiment-tasks` 改为以批次/项目为入口 + 叶子多选；`createExperimentTaskSchema`、`getTaskSampleOptions`、`experiment-task-form` 同步改。
- **样本级子驱动改扇出**：开始实验/提交反馈时，对该 task 经 TaskSample 关联的**每个叶子**扇出改状态（received/received_abnormal/waiting_task→lab_in_progress；→feedback_submitted），逐叶沿用现有源状态守卫（幂等：仅当叶子未进更后状态时改）。此口径与规格 §7.1「样本状态子驱动」一致（多样本下为近似、不影响项目聚合）。
- 产出指标随叶子（§2.3），任务只记"这次上机装了哪些样本"。

### 2.5 ExperimentRecordImage（实验记录图片，M3 用，M1 预留位）
| 字段 | 说明 |
|------|------|
| `id`/`projectId`/`batchId?`/`taskId` | 归属；**绑定到一次上机(ExperimentTask)**，其 TaskSample 集合即识别行可指派的候选叶子全集 |
| `blobPathname` | Vercel Blob 私有对象 pathname（**不是可直贴 URL**，读取见 §6） |
| `uploadedById`/`uploadedAt` | 上传人/时间 |
| `ocrStatus` | `pending`/`processing`/`done`/`failed`（重跑仅允许从 done/failed 触发） |
| `ocrModel` | 实际所用 `provider/model` 串（复算/成本审计；对齐"模型不写死"） |
| `ocrResult` | 结构化结果：`{ rows: [{ rawText?, sampleName, concentration, viability, aggregationRate }] }`；逐字段置信为**可选**（仅模型稳定返回时填、只用于高亮、不阻断） |
| `ocrError` | 失败原因（failed 时） |
| `confirmedById`/`confirmedAt` | 人工确认人/时间 |

**存证回链**：确认动作把 `imageId`（必要时含 `rowIndex`）写进 `operation_logs.afterData`（§6 step4 本就写操作日志，零新增结构）。仅当 M3 确需"按图直接反查落库记录"，再给 `QcRecord` 加可空 `sourceImageId` 外键。修正 v1 措辞——`source=ai_confirmed` 只是来源标记，图片溯源经操作日志/可选 FK，不是 source 枚举本身。**不加** sha256/byteSize/originalName/attempts（小团队照片上传无去重/重试诉求，KISS，将来需要再补）。

### 2.6 迁移与 demo 数据
- **不写保数据迁移脚本**：项目无 `prisma/migrations`、部署走 `prisma db push --accept-data-loss` + seed（见 `package.json` vercel-build）；且系绿地替代 Excel、扁平 Sample 仅承载过 demo。故 schema 变更由 **db push 同步**，demo 一律 **drop + reseed**。"正式数据一次性迁移"仅作终局附注：若将来有需保留数据，走独立一次性步骤（沿用 `predeploy.ts` 式前置 raw SQL 或临时切 prisma migrate），**必须在改 schema 的 db push 之前**完成，不进 vercel-build 流水线。
- **seed 改造**（与上一轮"经验挂业务项目"决策衔接，数据文件已对齐）：当前 `data/拜谱单细胞实验预约.xlsx` 与 `data/单细胞项目经验.xlsx`（**现名，无 `_2`；12 行/3 委托单，项目编号 BP-G260505743/505778/506038 全交集、每项目样本名数=预约样本数量 4/2/6**）已可 join。seed 简化为：预约表建项目 + 批次(sampleCount)，**经验表逐行直接建命名样本叶子**（按 projectNo + 样本名）+ 其 QC/产出，去掉"建 N 占位再匹配"中间步。join 不中时叶子产出留空、不报错。同步更新 `prisma/seed-experience.ts` 头注（现为"101 行/43 委托单/零交集"旧口径）。
- **字段迁移映射**（M1 显式）：旧 `Sample` 的 `batchNo(原 sampleNo)/sampleCount/到样接收字段` → `SampleBatch`；`remark` → 叶子；`QcRecord.sampleId` → 叶子、`taskId` 留作来源引用；产出指标从 `ExperimentTask` → 叶子。点名受影响领域逻辑：`ensureActualDateValid` 改读 `batch.receivedAt`（经 TaskSample→Sample→batch）、`receiveSample`、`lib/analytics`（§8）。

## 4. 交互模型（与现有 @sheet/Dialog 约定的对齐）

现有约定：查看=右侧滑 `DetailSheet`（@sheet 平行路由，URL 可分享）；创建=居中 `FormModal`；动作=居中 Dialog。设计稿把动作升级为右侧抽屉，但与 @sheet 同占右侧会层叠打架。**落点规则（M2 遵循、M4 回写 CLAUDE.md 时具体点名取代旧"动作一律 Dialog"）**：

- 查看详情 → 右侧滑 `DetailSheet`（不变，@sheet 专属）。
- 创建实体 → 居中 `FormModal`（不变）。
- **动作抽屉 = 本地组件态、不进 @sheet 路由**（动作无需深链/分享）。
- **杜绝双右浮层**：信息量大的队列动作（登记接收 ReceiveSheet、录反馈·质控 FeedbackSheet）→ 右侧动作抽屉，**仅从列表行触发**；从**详情侧滑内部**触发的动作 → 用居中 Dialog/AlertDialog（Radix 居中层叠在右 Sheet 之上，现状已验证可用），或在 sheet-body 内联渲染表单段，**绝不开第二个右抽屉**。

- **更省的备选（推荐先评估）**：保持现状"确认/破坏性动作=居中 Dialog"不动，只把收样/反馈两张高密度表单升级为抽屉、且只从列表行触发——天然规避双右浮层、改动面最小。

## 5. 进度式信息收集与状态机

原则：**字段可暂空，状态机只在"推进到下一棒"时校验该棒必需字段**，不要求一次填全。

- **建项目**：仅客户 + 服务档次必填；合同号/项目编号/样本编号/样本名/数量均可空。同事务建 **1 个空 SampleBatch**（batchNo/count 可空、0 叶子、status=waiting_arrival、列表显示"未编号待接收批次"）。**WHY**（承接现有"建项目即建 1 条待到样样本"的队列驱动接缝契约）：收样工位的接缝队列需要一行可显示的待接收批次，对称于 projectNo-null 草稿。
- **确认项目**（PM）：录入项目编号(委托单号)并校验唯一 → `waiting_sample`。
- **收样**（收样员）：登记接收时补 批次编号/物种/组织/运输 + **必填 count≥1**、据 count **建/订正 ≥1 叶子**（堵死"批次 received 但 0 叶子→无任务可建→项目永卡 sample_received"的状态空洞）。样本名此时可由任务单回填或留空。同事务 batch→received + 扇出叶子→received。项目聚合 `waiting_sample→sample_received` 触发点从"首个样本"改为"**首个批次接收成功**"（本期 1 批次/项目，等价）。
- **样本名录入路径**：**M2 即提供手工录入入口**（实验工位的样本命名抽屉或叶子编辑），不依赖 M3 多模态——否则 M2~M3 间经验视图归属（按 projectNo+样本名）断链。多模态(M3)只是把同一入口的值预填。建议 sampleName 在"建任务/录产出指标前"必填（否则经验归属无键）。
- **实验**（实验员）：排期 → 上机 → 上传实验记录图片 → 多模态识别 → 人工确认写回叶子 → 录反馈/产出。`recordRunMetrics`/`recordTaskQc` 改为按 (task, 叶子) 粒度，前置态=该叶子所在 task `completed`、可订正；`recordQcSchema` 增 `sampleId`，不再从 `before.sampleId` 推导；生信任务详情的产出录入挂件（`bioinfoExperimentTaskSelect` + `run-metrics-section`）从单值改为经 TaskSample 枚举叶子**逐行**。
- **交付**：按 `serviceLevel` 分两路径（qc/library/run 实验完成直接待交付；standard/advanced 经生信），保持聚合驱动。
- **项目时间线**：`operation_logs.entity_type` 增 `sample_batch`（收样日志挂批次）、`experiment_record_image`（M3 上传/确认）；时间线汇总来源相应扩展，否则收样/识别确认等关键事件不显示。

## 6. 多模态识别功能（M3，M1 预留位）

数据流（修订 v1 的私有 Blob/Server Action 隐患）：
1. 实验员在「实验」工位某次上机上点"上传实验记录" → **走 Route Handler**（multipart/stream，**不用 Server Action**——其请求体默认 1MB 会拒手机照片）；客户端**上传前下采样**（长边 ~1500px、压 JPEG），降体积与识别 token 成本。
2. 上传 handler 拿到 bytes 后**一次扇出**：并行 `put` 到 **Vercel Blob 私有**（access:'private'，存证）+ 以 **base64 inline** 传给模型识别——避免"先存私有再下载一跳"（私有对象模型取不到、也不能直贴 `<img>`）。建 `ExperimentRecordImage(ocrStatus=processing)`。
3. 经 **Vercel AI Gateway** 调 Claude 多模态，**单一实现**：AI SDK `generateObject` + zod schema（与项目已用 zod 一致、产出 TS 类型），同一 schema 约束模型输出与表单校验。模型用 `provider/model` 串由 `AI_OCR_MODEL` 注入（默认最新 Claude 多模态）。
4. 结果写 `ocrResult.rows` → **对齐步骤（核心，必做）**：确认抽屉渲染该 task 的候选叶子（按创建序/seq）；预绑定策略=先按已有 `sampleName` 精确匹配、匹配不上回退行序；每识别行带**可编辑样本名** + **"指派到哪条叶子(sampleId)"下拉**可重指派；**显式处理行数≠叶子数**（多/少于候选叶子→高亮提示、人工增删/忽略，**禁止静默自动增删叶子**）。
5. 实验员核对/修正 → 提交 → 按"行→选定 sampleId"**成对写** `Sample.sampleName` + `QcRecord`（校验每行绑定到唯一且 ∈该 task TaskSample 的 sampleId，未绑定不许提交）；标 `source=ai_confirmed`、记 `confirmedBy/At`、写操作日志（含 imageId+rowIndex 存证）。
6. **原图查看**走鉴权代理 Route Handler（校验当前用户在该项目 scope 内 → 服务端带 token 下载流式回传），存 pathname 而非可直贴 URL。

**根因辅助**：在收样/确认项目时从权威任务单（如 BP-G260506372 的 4 个报告组别名）回填叶子 `sampleName`，使首轮即可按名精确匹配，降低对行序依赖。

约束：**人在回路**（仅预填、不自动落库）；**幂等/并发**（见 §9）；**失败兜底**（失败/低置信 → 退回手工，不阻断）；**模型不写死**。

## 6.5 生信协作（产出 API 上报 · 任务单生成 · 邮件提醒）

> 2026-06 老板新增。生信分析在独立系统执行，本平台需与之协作：接收产出、下发任务单、提醒。三者均在 M3。

**产出指标 API 上报（机对机）**：部分产出指标（基因中位数 `medianGenes`、捕获细胞数 `capturedCells`、测序量、`suspensionType`）由 cellranger 在生信系统侧产出，支持生信系统经 **API 上报**回写本平台，而非只靠人工 UI。
- 端点：`POST /api/integrations/run-metrics`（机对机），**服务令牌鉴权**（API key / 服务账号，非 NextAuth 会话；新增 env `INTEGRATION_API_KEY` 或服务账号记录，校验后以系统身份写、operator=服务账号）。
- 归属键：`projectNo + sampleName`（同经验视图 join 键）定位样本叶子；写产出字段、可订正、写操作日志。
- 与既有路径并存：实验员 UI / 多模态识别仍可手工录/订正（多模态主要补 QC：浓度/活率/结团率 + 样本名）；**API 上报是生信侧产出主路径**。三路同写同一叶子产出，统一幂等（按叶子 upsert，见 §9）。

**生信任务单生成**：按 `docs/单细胞任务单-BP-G260506372.md` 的结构，从项目数据生成生信任务单（项目信息 / 样品明细 / 物种 / 实验设计 / 差异比较组等），供生信执行。
- 数据来源：Project + SampleBatch + Sample 叶子（样本名/物种/组织）+ 服务档次/平台；样品明细 = 逐叶子。
- 形态：服务端按代码内模板渲染结构化文档（MD/HTML，可下载、可作邮件附件），**不新建实体**（按需从数据渲染）。

**邮件任务提醒**：生信任务创建/指派时、或待处理任务提醒，发邮件通知相关人（可附任务单）。
- **属第一期范围的显式增量**（规格 §13 原排除"通知"）：仅做"任务提醒"这一最小必需，**不做**通用通知中心/站内信/铃铛（与工位制"看得见喊得着"一致）。
- 基础设施：邮件服务（Resend / SMTP，env 配置，`NOTIFY_ENABLED` 可关）；发送记录写操作日志；失败不阻断业务动作。

## 9. 风险与开放项

- **行级可见范围（安全横切）**：role-scope 重写后必须用例验证不 fail-open（各角色可见集正确、角标按批次而非样本计数）；列表/详情/导出/统计/角标五处口径同步。
- **多模态幂等/并发（M3）**：人工确认写 QC 须幂等（按叶子 upsert / 先查后写，**别复用恒 create 的 `recordTaskQc`**）；`confirmedAt` 已写则拦截二次确认、需显式"重新确认"覆盖并记日志；OCR 重跑仅从 done/failed 触发。图片内容去重（省配额）列为可选成本优化。
- **隐私/数据出境（M3，重点）**：图片可能含患者/客户信息，识别要把整图发给境外 Claude（经 AI Gateway）——属 PII 出境。需：明确数据出境与处理方、与客户的告知/授权口径；对该路径**关闭/零留存 Gateway 请求日志**；评估上传前对无关 PII 区域裁剪；预留 provider 抽象（`provider/model` 串）以便将来切境内多模态模型。
- **依赖与环境变量（M3）**：待加 `ai` + `@ai-sdk` gateway provider（或 `@anthropic-ai/sdk`）+ `@vercel/blob`；env：`AI_GATEWAY_API_KEY`（Docker/Vercel 通用，OIDC 仅 Vercel 自动；**修正 v1"部署在 Vercel"的隐含绑定**——本仓库 Docker + Vercel 双轨）、`BLOB_READ_WRITE_TOKEN`（自托管必需）、`MULTIMODAL_ENABLED`（开关，关则隐藏入口走纯手工）、`AI_OCR_MODEL`。
- **第一期范围边界**：规格 §13 排除"大文件数据交付和云存储"、"通知"，全文无 AI/多模态。需澄清并写清：**实验记录图片存证（小图、私有）≠ §13#7 大文件数据交付**；多模态识别 + AI Gateway、**邮件任务提醒**（仅最小任务提醒、非通用通知中心）均为对第一期范围的**显式增量**（在规格 §6 新增 AI/协作模块章节或 §13 加注解）。
- **生信集成鉴权（M3）**：产出 API 上报走服务令牌（`INTEGRATION_API_KEY` 或服务账号），非 NextAuth；写操作日志 operator=服务账号；与 UI/多模态三路同写产出须统一幂等（按叶子 upsert）。
