# 单细胞云平台 - 需求规格说明书

## 1. 产品定位

**产品名称**：单细胞云平台

**产品定位**：面向单细胞实验服务业务的项目管理、样本流转、实验任务、质控记录、生信分析、经验沉淀与交付跟踪平台。

系统不应只理解为“预约平台”。旧 Excel 中的“预约”本质是人工维护项目流程信息，包括客户取样、预计到样、样本接收、实验安排、上机反馈和结果说明。有系统之后，预约应转化为平台中的**任务生成、排期确认、状态流转、结果反馈和交付闭环**。

## 2. 建设目标

### 2.1 完整目标

单细胞云平台最终应覆盖从销售发起项目到生信交付结果的完整服务链路：

```text
销售负责人创建项目
-> 项目经理确认需求与流转计划
-> 样本接收员登记到样与异常
-> 实验执行员排期、执行实验并录入质控结果
-> 生信分析员接收数据并推进分析任务
-> 项目经理确认交付并关闭项目
```

### 2.2 第一期 MVP 目标

第一期目标是替代当前 Excel 管理方式，先跑通核心闭环：

1. 建立项目、样本、实验任务、质控记录、生信分析任务和经验库数据模型。
2. 支持按角色推进项目流程，而不是只维护一张预约表。
3. 支持从现有 Excel 初始化导入历史数据。
4. 支持项目详情页查看完整时间线。
5. 支持基础统计图表和筛选导出。
6. 不引入复杂工作流引擎，先用状态机、动作 API、数据库事务和操作日志管理流程。

| 角色 | 可见范围 where |

| 管理员 / 项目经理 | 无过滤（全部） |

## 4. 核心业务流程

### 4.1 项目全流程

项目主干分两条交付路径，由「服务档次」（§6.3 `service_level`）决定：多数单子做完实验走生信分析后交付，少数只做到实验/质控/建库即交付。两条路径都汇入「待交付」，统一由项目经理确认交付。

```text
项目草稿 -> 项目已确认 -> 待到样 -> 已到样 -> 实验中
实验中（「提交实验反馈」时按服务档次分流）：
  ├─ 含生信分析的服务  -> 待生信分析 -> 生信分析中 -> 待交付
  └─ 不含生信的服务（质控版/建库版/上机版）-> 待交付
待交付 -> 已交付 -> 已完成
```

说明：

1. 分流发生在「提交实验反馈」动作，依据项目 `service_level`。质控是交付前的常规环节，质控结论即质控版项目的交付物；**不做生信不等于异常**，成功的非分析交付不得走异常路径（否则污染异常统计）。
2. 项目中间状态不由用户直接设置，而是由子实体（样本/实验任务/生信任务）的动作在同一事务内聚合驱动，规则见 §7.1。

异常路径：

```text
任意未完成状态 -> 项目异常 -> 恢复至异常前状态 / 异常终止
```

「项目异常」记录异常前状态（`status_before_abnormal`），「恢复」回到该状态，「异常终止」（`terminated`）为终态。对应动作与端点见 §5.1 / §10。注意区分两类「无分析交付」：质控版正常交付走主干到 `completed`；样本全部不满足上机要求、无有效交付物的，走「异常终止」。

### 4.2 角色协作流程

1. **销售负责人**创建项目，填写客户单位、客户联系人、合同编号、委托单编号、销售负责人、预计到样信息和服务类型。
2. **项目经理**确认项目信息，补充项目经理、项目优先级、实验需求、交付要求，并将项目推进到待到样。
3. **样本接收员**登记样本实际到样时间、运输条件、样本数量和异常情况。
4. **项目经理或实验执行员**根据已接收样本创建实验任务。
5. **实验执行员**设置计划实验日期、负责人、实验类型和上机方式。
6. **实验执行员**开始实验并录入质控数据，包括悬液浓度、活率、结团率、质控结论和风险等级。
7. **实验执行员**录入上机样本个数和结果反馈。
8. **生信分析员**接收生信分析任务，更新分析状态、报告状态和交付日期。
9. **项目经理**确认交付完成，关闭项目。

## 5. 流程管理实现原则

第一期不使用 Hatchet。流程管理通过以下机制实现：

1. **状态枚举**：项目、样本、实验任务、生信任务分别有明确状态字段。
2. **动作 API**：用户不直接随意修改状态，而是执行“确认项目”“接收样本”“设置排期”“开始实验”“录入质控”“提交反馈”“创建生信任务”“确认交付”等动作。
3. **状态转移校验**：每个动作校验当前状态、角色权限和必填字段。
4. **数据库事务**：跨表更新必须在 Prisma transaction 中完成。
5. **操作日志**：所有关键动作写入 `operation_logs`。
6. **项目时间线**：从操作日志和业务记录中汇总项目全流程事件。

### 5.1 关键动作与状态转移

下表是系统的**权威动作集**（非穷举示例）：用户只能通过这些动作改变状态，每个动作在 §10 有对应端点，并按需触发 §7.1 的项目状态聚合。

| 动作 | 发起角色 | 前置状态 | 结果 |
|------|----------|----------|------|
| 创建项目 | 销售负责人/项目经理 | 无 | 项目草稿 |
| 确认项目 | 项目经理 | 项目草稿 | 项目已确认（补充项目经理、服务档次等） |
| 标记待到样 | 项目经理 | 项目已确认 | 待到样 |
| 接收样本 | 样本接收员 | 待到样 | 样本已接收（首个样本触发项目→已到样） |
| 登记样本异常 | 样本接收员 | 待到样/已接收 | 样本异常 |
| 创建实验任务 | 项目经理/实验执行员 | 已到样 | 任务待排期 |
| 设置排期 | 实验执行员 | 待排期/已排期 | 任务已排期（设计计划日期，可同时指派负责人/上机方式/部门） |
| 开始实验 | 实验执行员 | 已排期 | 任务进行中（首个任务触发项目→实验中；同事务内样本→实验中） |
| 录入质控 | 实验执行员 | 进行中/待反馈 | 创建质控记录（业务判定，不改任务执行态；一任务可多条） |
| 完成实验（待反馈） | 实验执行员 | 进行中 | 任务待反馈（台面操作完成、等待结果录入；可录上机个数）。可选步骤，亦可由「进行中」直接提交反馈 |
| 提交实验反馈 | 实验执行员 | 进行中/待反馈 | 任务已完成（结果状态+反馈必填）；同事务内样本→已反馈；项目下任务全部完成时按服务档次分流（→待生信分析 或 →待交付） |
| 创建生信任务 | 项目经理/实验执行员/生信分析员 | 实验任务已完成且服务含生信 | 生信任务待开始 |
| 开始分析 | 生信分析员 | 待开始 | 分析中（首个生信任务触发项目→生信分析中；同事务内可填数据接收日期、指派分析员） |
| 提交审核 | 生信分析员 | 分析中 | 待审核（分析完成、自检报告）。可选步骤，亦可由「分析中」直接提交报告 |
| 提交报告 | 生信分析员 | 分析中/待审核 | 生信待交付（记交付物说明；项目下生信任务全部待交付时项目→待交付） |
| 确认交付 | 项目经理 | 待交付 | 已交付（同事务内级联项目下「待交付」生信任务→已交付并落报告交付日期，§8.9） |
| 完成项目 | 项目经理 | 已交付 | 已完成 |
| 标记异常 | 项目经理/管理员 | 任意未完成态 | 项目异常（记录异常前状态） |
| 恢复异常 | 项目经理/管理员 | 项目异常 | 恢复至异常前状态 |
| 异常终止 | 项目经理/管理员 | 项目异常/任意未完成态 | 异常终止（终态） |

## 6. 功能模块

### 6.1 登录与用户管理

MVP 功能：

1. 用户登录和退出。
2. 管理员创建用户。
3. 管理员启用、禁用用户。
4. 管理员设置用户角色。
5. 用户可修改自己的密码。
6. 管理员重置用户密码（第一期无自助找回通道，管理员重置是忘记密码的唯一恢复路径）。
7. 默认初始化管理员账号。

不建议第一期开放自助注册。

### 6.2 首页仪表盘

首页用于让不同角色快速看到待处理事项。

通用指标：

1. 项目总数。
2. 待到样项目数。
3. 今日/本周预计到样样本数。
4. 待排期实验任务数。
5. 进行中实验任务数。
6. 待生信分析任务数。
7. 生信分析中任务数。
8. 待交付项目数。
9. 异常项目数。

角色待办：

| 角色 | 待办示例 |
|------|----------|
| 销售负责人 | 待补充合同/委托单信息、自己负责项目状态 |
| 项目经理 | 待确认项目、异常项目、待交付项目 |
| 样本接收员 | 今日预计到样、待接收样本 |
| 实验执行员 | 待排期任务、今日实验任务、待反馈任务 |
| 生信分析员 | 待开始分析、分析中、待提交报告 |

### 6.3 项目管理

项目是平台的主业务对象。

核心字段：

| 字段 | 说明 |
|------|------|
| 项目编号 | 项目业务键，对齐**委托单编号**粒度（一委托单=一项目）；导入无来源时系统生成，规则见 §15 |
| 合同编号 | 合同标识，**可重复**（一个合同可对应多个项目/委托单） |
| 委托单编号 | 委托单标识，项目的天然业务键 |
| 客户单位 | 客户机构或公司 |
| 客户姓名 | 客户联系人 |
| 联系方式 | 电话、邮箱等 |
| 销售负责人 | 关联用户 |
| 项目经理 | 关联用户 |
| 项目类型 | 单细胞转录组、免疫组、空间转录组等 |
| 服务内容 | `service_items`，具体服务项明细（文本/多选），承接旧表「上机方式」长文本中的服务说明 |
| 服务档次 | `service_level` 枚举，**驱动流程分支**：`qc`(质控版) / `library`(建库版) / `run`(上机版) / `standard`(标准分析版) / `advanced`(高级分析版) |
| 测序平台 | `sequencing_platform`，如 Illumina / 真迈，承接旧表「上机方式」中的平台信息 |
| 项目优先级 | 普通、加急 |
| 项目状态 | 状态枚举 |
| 预计交付日期 | 项目经理维护 |
| 实际交付日期 | 交付时记录 |
| 备注 | 补充说明 |

MVP 功能：

1. 项目列表、搜索、筛选、排序、分页。
2. 新增、编辑项目。
3. 项目详情页。
4. 项目状态动作按钮。
5. 关联样本、实验任务、生信任务、质控记录和操作时间线。
6. 项目异常标记和异常说明。

**服务档次与流程分流**：`standard`/`advanced` 含生信分析，走「待生信分析」分支；`qc`/`library`/`run` 不含生信，实验反馈后直接进「待交付」。`service_level` 在创建或确认项目时确定，历史数据可由旧表「上机方式」文本解析得到（§15）。

### 6.4 样本管理

样本是实验流转的核心对象。

核心字段：

| 字段 | 说明 |
|------|------|

5. 从样本创建实验任务。

### 6.5 实验任务管理

实验任务承载实验室排期和执行过程。

核心字段：

| 字段 | 说明 |
|------|------|
| 任务编号 | 系统生成 |
| 关联项目 | 项目 |
| 关联样本 | 一个或多个样本 |
| 实验类型 | 组织解离、建库、上机、质控等 |
| 上机方式 | 10x、GEM-X、Next GEM、墨卓等 |
| 计划实验日期 | 实验执行员设置 |
| 实际实验日期 | 实际执行日期 |
| 实验负责人 | 实验执行员 |
| 所属部门 | 如时空部、单细胞部等 |
| 上机样本个数 | 实际上机数量 |
| 任务状态 | 状态枚举 |
| 结果状态 | 正常上机、低风险上机、不满足要求、其他 |
| 结果反馈 | 文本说明 |

MVP 功能：

1. 任务列表和待排期任务池。
2. 设置排期和负责人。

## 7. 状态设计

### 7.1 项目状态

| 状态 | 代码 | 说明 |
|------|------|------|
| 项目草稿 | `draft` | 销售或项目经理创建，信息未确认 |
| 项目已确认 | `confirmed` | 项目经理确认项目需求 |
| 待到样 | `waiting_sample` | 等待客户样本到达 |
| 已到样 | `sample_received` | 至少一个样本已接收 |
| 实验中 | `lab_in_progress` | 已进入实验任务 |
| 待生信分析 | `waiting_bioinfo` | 实验完成，等待生信 |
| 生信分析中 | `bioinfo_in_progress` | 生信分析进行中 |
| 待交付 | `waiting_delivery` | 报告或结果待交付确认 |
| 已交付 | `delivered` | 已交付结果 |
| 已完成 | `completed` | 项目关闭 |
| 项目异常 | `abnormal` | 发生异常，待处理；记录 `status_before_abnormal` 以便恢复 |
| 异常终止 | `terminated` | 项目无法继续，终态 |

**项目状态聚合驱动规则**：项目中间状态不由用户直接设置，而由下列子实体动作在同一 Prisma 事务内联动改写并记入 `operation_logs`。

| 触发动作（子实体） | 项目状态转移 |
|------|------|
| 接收样本（项目下首个样本接收成功） | `waiting_sample` → `sample_received` |
| 开始实验（项目下首个实验任务 start） | `sample_received` → `lab_in_progress` |
| 提交实验反馈（项目下实验任务全部完成）+ 服务含生信 | `lab_in_progress` → `waiting_bioinfo` |
| 提交实验反馈（全部完成）+ 服务不含生信 | `lab_in_progress` → `waiting_delivery` |
| 开始分析（项目下首个生信任务 start） | `waiting_bioinfo` → `bioinfo_in_progress` |
| 提交报告（项目下生信任务全部进入待交付） | `bioinfo_in_progress` → `waiting_delivery` |
| 确认交付 | `waiting_delivery` → `delivered`（→ `completed`，见 §5.1） |

「全部完成/全部进入」以项目下子实体集合为判定，单条子实体动作不直接把项目跳到终态。

**样本状态的子驱动**（与上表同一事务内联动，承接 §7.2 的 `lab_in_progress`/`feedback_submitted`）：开始实验时样本 `received`/`received_abnormal`/`waiting_task` → `lab_in_progress`；提交实验反馈时样本 → `feedback_submitted`。MVP 单 `sample_id` 任务下为 1:1 精确；多任务/样本（终局 `task_samples`）下为近似，不影响项目聚合判定。

**生信任务的交付级联**（与「确认交付」同一事务内联动，承接 §7.4 的 `delivered` 与 §8.9）：项目经理确认交付时，项目下处于「待交付」的生信任务统一 → `delivered` 并落 `deliveredAt` = 项目交付日期。生信任务的 `delivered` 不由独立动作设置。

## 8. 数据校验规则

> 总则：所有跨字段的**日期比较一律按自然日**（DATE 截断后比较），同日视为通过。`received_at` 等保留时分仅用于展示/排序，不参与日期先后判定。

1. **样品编号 `sample_no` 在业务样本表内唯一**；经验库 `experience_records` 不受此约束（历史经验天然一项目多条）。
2. **项目编号唯一性**：`projects.project_no`（=委托单号粒度）在业务项目表内唯一，合同号 `contract_no` 可重复。**注意 `experience_records.project_no` 实为委托单号、天然非唯一**，不施加唯一约束（见 §9.7 / §15）。历史导入无法保证唯一时保留原编号并生成内部 ID。
3. 样本数量、上机样本个数、测序量、捕获细胞数、基因中位数必须为非负整数（**允许为空**，未填不等于 0）。
4. 活率和结团率必须为 0-100 之间的数字（仅对结构化数值列校验；预约表埋在自由文本里的活率不在此列，见 §15）。
5. 日期字段必须为有效日期。
6. 预计到样日期不能早于客户取样日期（同日通过）。
7. 实际实验日期的自然日不能早于样本接收日期的自然日（**同日通过**）；仅当实验自然日真正早于接收自然日时，才需管理员确认异常录入。
8. 实验任务完成时必须填写结果状态（`result_status` 枚举）和结果反馈（`result_feedback` 文本）。历史导入时 `result_status` 由「结果反馈」文本启发式映射、无法判定时兜底为「其他」，原文全量保留（§15）。
9. 生信任务已交付时必须填写报告交付日期。
10. 关键状态变更必须有操作人和操作日志；历史导入动作的操作类型为 `import`、操作人为导入执行者（§6.10）。

## 15. 导入映射与清洗规范

本章把 §6.9 / §14.8 / §14.9 的「从 Excel 初始化」落成逐列可执行规则，是导入功能的实现依据。

### 15.1 通用原则

1. 历史导入走独立路径（`/api/import/*`），**直写终态、不经动作 API 的前置状态校验**；仍写 `operation_logs(action=import)`（§6.10）。
2. **原文零丢失**：无结构化落点的原始文本保留到对应 raw 字段或 `remark`。
3. **失败行不中断整批**：未通过校验的行记入 `import_logs.error_detail` 供人工复核（§9.9）。
4. 导入前经 `/api/import/preview` 做字段校验与预览，重复样品按 §6.9 跳过/覆盖。

### 15.2 预约表 → 项目 / 样本 / 实验任务（逐列映射）

| 源列（预约表） | 目标 | 处理规则 |
|------|------|------|
| 委托单编号 | projects.order_no / project_no | **分组键**：同委托单号的多行合并为一个项目；project_no 取委托单号 |
| 合同编号* | projects.contract_no | 去星号；可重复 |
| 销售负责人 | projects.sales_owner_id | 按 name 匹配 users（见 15.7） |
| 客户单位 / 客户姓名 | projects.customer_org / customer_name | 直映 |
| 样品编号 | samples.sample_no | 唯一键；重复按 §6.9 跳过/覆盖 |
| 样本物种 / 组织类型 / 数量 | samples.species / tissue_type / sample_count | 直映；物种、组织为开放词表，不硬枚举 |
| 实验类型 | samples.experiment_type | 直映（如「组织解离」） |
| 运输条件 | samples.transport_condition | 「冷链4度」「4度」归一为标准词，保留原文 |
| 客户取样 / 预计到样日期 | samples.sampling_date / expected_arrival_date | 日期解析 |
| 上机方式 | run_method + service_level + sequencing_platform + service_items；原文存 raw_booking_text | **解析见 15.4** |
| 样本接收日期 | samples.received_at | 含时分，保留 |
| 实验室预约日期 | experiment_tasks.planned_date | 计划实验日期 |
| 实验日期 | experiment_tasks.actual_date | 实际实验日期 |
| 上机样本个数 | experiment_tasks.loaded_sample_count | 非负整数；0 表示未上机 |
| 结果反馈 | result_feedback（原文）+ result_status（启发式） | **映射见 15.5** |
| 填表日期 | projects.created_at | 预约填报时间 |
| 备注 / 备注2 | remark | 合并 |

### 15.3 导入目标状态映射（直写终态）

| 条件 | experiment_task | sample | project |
|------|------|------|------|
| 结果反馈含「正常上机」、上机个数>0 | completed | feedback_submitted | 含生信→waiting_bioinfo；不含生信→completed |
| 部分正常 + 部分低风险/异常 | completed | feedback_submitted | 同上 |
| 上机个数=0 或 含「不满足上机要求」 | completed（result=不满足要求） | abnormal | 无有效交付→terminated |

说明：历史数据无生信记录，`standard`/`advanced` 项目导入后落 `waiting_bioinfo` 待补录；`qc`/`library`/`run` 项目直接 `completed`。具体终态有歧义的行，按本表默认值落库并可由管理员事后调整。

### 15.4 「上机方式」文本解析

先归一化（大小写、去换行），再派生多字段，解析失败降级为「仅存 raw_booking_text + 人工复核」，不丢数据：

- `run_method`：匹配 NEXT-GEM / GEM-X / Next GEM / 墨卓 等建库化学关键词
- `project_type`：匹配 转录组 / 免疫组（含 TCR）/ 空间转录组
- `service_level`：「质控版」→qc；「做到建库」→library；「标准分析版」→standard；「高级」→advanced；仅上机→run
- `sequencing_platform`：括号内 Illumina / 真迈
- 「做到质控/建库，不上机」→ 对应 service_level + 标记不上机

### 15.5 「结果反馈」→ result_status 启发式

优先级：含「不满足/不满足上机」→不满足要求；同时含「正常上机」与其他结论（低风险/偏黄/凋亡/不满足）→其他；纯「正常上机」→正常上机；仅含「低风险」→低风险上机；无法判定→其他。「冰袋/碎冰/到样/冷链」类识别为到样/运输信息，写 samples.transport_condition 或 abnormal_note，不进 result_status。**预约表导入不回填 qc_records**（活率等仅存于 result_feedback 文本），结构化质控数值由经验库导入或后续手工补录。

### 15.6 经验表 → experience_records

| 源列 | 目标 | 规则 |
|------|------|------|
| 项目编号 | project_no | 委托单号格式，**非唯一，不去重** |
| 物种 / 部位 / 项目类型 / 细胞核 | 对应列 | **原样导入**（部位脏数据保留，规范化清洗推后——老板已确认）；「大鼠」等为开放词表 |
| 悬液浓度 / 活率 / 结团率 / 测序量 / 捕获细胞数 / 基因中位数 | 对应数值列 | 数值校验（活率、结团率 0-100） |
| 备注 | remark | 旧表此列存在错位（含项目编号/试剂说明），原样保留、不解析 |

重复导入按 (project_no + 行指纹) 跳过，避免翻倍；不设业务去重键（一项目多条为正常）。

### 15.7 人名 → users 匹配

1. 按 name 匹配 users；name 不保证唯一，导入预览阶段输出**人名映射表**供人工确认。
2. 未命中：建 `is_active=false`、无密码（不可登录）的占位用户，role 按列含义（如 sales_owner），记入 `import_logs.error_detail`，管理员后续补全或合并。
3. 无源角色（项目经理 / 接收员 / 实验 / 生信负责人）导入留空，由后续业务动作赋值。
