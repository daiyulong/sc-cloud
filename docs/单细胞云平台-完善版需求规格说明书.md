# 单细胞云平台 - 需求规格说明书

## 1. 产品定位

**产品名称**：单细胞云平台

**产品定位**：面向单细胞实验服务业务的项目管理、样本流转、实验任务、质控记录、生信分析、经验沉淀与交付跟踪平台。

系统不应只理解为“预约平台”。旧 Excel 中的“预约”本质是人工维护项目流程信息，包括客户取样、预计到样、样本接收、实验安排、上机反馈和结果说明。有系统之后，预约应转化为平台中的**任务生成、排期确认、状态流转、结果反馈和交付闭环**。

## 2. 建设目标

### 2.1 完整目标

单细胞云平台最终应覆盖从销售发起项目到生信交付结果的完整服务链路：

```text
销售负责人创建项目
-> 项目经理确认需求与流转计划
-> 样本接收员登记到样与异常
-> 实验执行员排期、执行实验并录入质控结果
-> 生信分析员接收数据并推进分析任务
-> 项目经理确认交付并关闭项目
```

### 2.2 第一期 MVP 目标

第一期目标是替代当前 Excel 管理方式，先跑通核心闭环：

1. 建立项目、样本、实验任务、质控记录、生信分析任务数据模型（经验数据并入实验任务，无独立经验表，§9.7）。
2. 支持按角色推进项目流程，而不是只维护一张预约表。
3. 支持从现有 Excel 初始化导入历史数据。
4. 支持项目详情页查看完整时间线。
5. 支持基础统计图表和筛选导出。
6. 不引入复杂工作流引擎，先用状态机、动作 API、数据库事务和操作日志管理流程。

### 3.3 数据可见范围（行级 scope）

权限矩阵「查看」列定义的行级可见范围，由服务端在所有列表/详情/导出/统计查询中**自动注入 Prisma where**（与用户筛选条件 AND 叠加），不依赖前端控制：

| 角色 | 可见范围 where |
|------|------|
| 管理员 / 项目经理 | 无过滤（全部） |
| 销售负责人 | `{ sales_owner_id: me }` |
| 样本接收员 | `{ samples: { some: { receiver_id: me } } }` |
| 实验执行员 | `{ experiment_tasks: { some: { operator_id: me } } }` |
| 生信分析员 | `{ bioinfo_tasks: { some: { analyst_id: me } } }` |
| 只读用户 | 授权范围，见下 |

- **「池」例外（鸡生蛋）**：实体在被指派给某人之前，按"我相关"过滤会恒假，导致该动手的人看不到待处理项。故下列角色的 scope 在自身相关项之外**并上对应待处理池**：实验执行员 += 全部待排期任务（`waiting_schedule`）∪ 待建实验任务的已收样本（已接收/待实验、所属项目在可建态、尚无实验任务）；生信分析员 += 全部待开始生信任务（`pending`）∪ 待建生信的完成实验任务（已完成、项目含生信且在可建态、尚无生信任务）；样本接收员 += 全部待到样样本（`waiting_arrival`）。这些池支撑工作台的「待建实验任务/待建生信任务/待到样」队列与对应列表 `awaiting`/`received` 筛选，口径三处共用一份 where 片段。
- viewer「授权范围」用显式授权关系表 `project_grants(user_id, project_id)`，未授权不可见。
- **历史导入项目**的 receiver/operator/analyst 多为空（旧表无此信息），对这些角色「相关」判定恒假——导入项目默认对管理员/项目经理可见，并提供关联回填入口；其余角色补录关联后可见。
- 第一期若资源有限，至少实现 admin/项目经理（全部）与 销售（自己负责）两档，其余档保留设计、按优先级补，不留白。

## 4. 核心业务流程

### 4.1 项目全流程

项目主干分两条交付路径，由「服务档次」（§6.3 `service_level`）决定：多数单子做完实验走生信分析后交付，少数只做到实验/质控/建库即交付。两条路径都汇入「待交付」，统一由项目经理确认交付。

```text
项目草稿 -> 待到样 -> 已到样 -> 实验中
实验中（「提交实验反馈」时按服务档次分流）：
  ├─ 含生信分析的服务  -> 待生信分析 -> 生信分析中 -> 待交付
  └─ 不含生信的服务（质控版/建库版/上机版）-> 待交付
待交付 -> 已完成（确认交付即关闭项目）
```

说明：

1. 分流发生在「提交实验反馈」动作，依据项目 `service_level`。质控是交付前的常规环节，质控结论即质控版项目的交付物；**不做生信不等于异常**，成功的非分析交付不得走异常路径（否则污染异常统计）。
2. 项目中间状态不由用户直接设置，而是由子实体（样本/实验任务/生信任务）的动作在同一事务内聚合驱动，规则见 §7.1。

异常路径：

```text
任意未完成状态 -> 项目异常 -> 恢复至异常前状态 / 异常终止
```

「项目异常」记录异常前状态（`status_before_abnormal`），「恢复」回到该状态，「异常终止」（`terminated`）为终态。对应动作与端点见 §5.1 / §10。注意区分两类「无分析交付」：质控版正常交付走主干到 `completed`；样本全部不满足上机要求、无有效交付物的，走「异常终止」。

### 4.2 角色协作流程

1. **销售负责人**创建项目，填写客户单位、客户联系人、合同编号、委托单编号、销售负责人、预计到样信息和服务类型。
2. **项目经理**确认项目信息，补充项目经理、项目优先级、实验需求、交付要求，并将项目推进到待到样。
3. **样本接收员**登记样本实际到样时间、运输条件、样本数量和异常情况。
4. **项目经理或实验执行员**根据已接收样本创建实验任务。
5. **实验执行员**设置计划实验日期、负责人、实验类型和上机方式。
6. **实验执行员**开始实验并录入质控数据，包括悬液浓度、活率、结团率、质控结论和风险等级。
7. **实验执行员**录入上机样本个数和结果反馈。
8. **生信分析员**接收生信分析任务，更新分析状态、报告状态和交付日期。
9. **项目经理**确认交付，项目随之进入已完成（关闭）。

## 5. 流程管理实现原则

第一期不使用 Hatchet。流程管理通过以下机制实现：

1. **状态枚举**：项目、样本、实验任务、生信任务分别有明确状态字段。
2. **动作 API**：用户不直接随意修改状态，而是执行“确认项目”“接收样本”“设置排期”“开始实验”“录入质控”“提交反馈”“创建生信任务”“确认交付”等动作。
3. **状态转移校验**：每个动作校验当前状态、角色权限和必填字段。
4. **数据库事务**：跨表更新必须在 Prisma transaction 中完成。
5. **操作日志**：所有关键动作写入 `operation_logs`。
6. **项目时间线**：从操作日志和业务记录中汇总项目全流程事件。

### 5.1 关键动作与状态转移

下表是系统的**权威动作集**（非穷举示例）：用户只能通过这些动作改变状态，每个动作在 §10 有对应端点，并按需触发 §7.1 的项目状态聚合。

| 动作 | 发起角色 | 前置状态 | 结果 |
|------|----------|----------|------|
| 创建项目 | 销售负责人/项目经理 | 无 | 项目草稿（暂无项目编号；同时录入样本编号+数量，生成 1 条待到样样本） |
| 确认项目 | 项目经理 | 项目草稿 | 待到样（补充项目经理、服务档次等；确认即进待到样，无独立中间态） |
| 登记接收 | 样本接收员 | 待到样 | 补全样本信息（物种/组织/实验类型/运输条件）+ 记录到样 → 已接收（首个样本触发项目→已到样） |
| 登记样本异常 | 样本接收员 | 待到样/已接收 | 样本异常 |
| 创建实验任务 | 项目经理/实验执行员 | 已到样 | 任务待排期 |
| 设置排期 | 实验执行员 | 待排期/已排期 | 任务已排期（设计计划日期，可同时指派负责人/上机方式/部门） |
| 开始实验 | 实验执行员 | 已排期 | 任务进行中（首个任务触发项目→实验中；同事务内样本→实验中） |
| 录入质控 | 实验执行员 | 进行中/待反馈/已完成 | 创建质控记录（业务判定，不改任务执行态；一任务可多条）。已完成为**补录**——漏填质控后提交反馈仍可事后补，不被卡死 |
| 完成实验（待反馈） | 实验执行员 | 进行中 | 任务待反馈（台面操作完成、等待结果录入；可录上机个数）。可选步骤，亦可由「进行中」直接提交反馈 |
| 提交实验反馈 | 实验执行员 | 进行中/待反馈 | 任务已完成（结果状态+反馈必填）；同事务内样本→已反馈；项目下任务全部完成时按服务档次分流（→待生信分析 或 →待交付） |
| 创建生信任务 | 项目经理/实验执行员/生信分析员 | 实验任务已完成且服务含生信 | 生信任务待开始 |
| 开始分析 | 生信分析员 | 待开始 | 分析中（首个生信任务触发项目→生信分析中；同事务内可填数据接收日期、指派分析员） |
| 提交审核 | 生信分析员 | 分析中 | 待审核（分析完成、自检报告）。可选步骤，亦可由「分析中」直接提交报告 |
| 提交报告 | 生信分析员 | 分析中/待审核 | 生信待交付（记交付物说明；项目下生信任务全部待交付时项目→待交付） |
| 确认交付 | 项目经理 | 待交付 | 已完成（终态；确认交付即关闭项目，同事务内级联项目下「待交付」生信任务→已交付并落报告交付日期，§8.9） |
| 标记异常 | 项目经理/管理员 | 任意未完成态 | 项目异常（记录异常前状态） |
| 恢复异常 | 项目经理/管理员 | 项目异常 | 恢复至异常前状态 |
| 异常终止 | 项目经理/管理员 | 项目异常/任意未完成态 | 异常终止（终态） |

## 6. 功能模块

### 6.1 登录与用户管理

MVP 功能：

1. 用户登录和退出。
2. 管理员创建用户。
3. 管理员启用、禁用用户。
4. 管理员设置用户角色。
5. 用户可修改自己的密码。
6. 管理员重置用户密码（第一期无自助找回通道，管理员重置是忘记密码的唯一恢复路径）。
7. 默认初始化管理员账号。

不建议第一期开放自助注册。

### 6.2 首页仪表盘

首页用于让不同角色快速看到待处理事项。

通用指标：

1. 项目总数。
2. 待到样项目数。
3. 今日/本周预计到样样本数。
4. 待排期实验任务数。
5. 进行中实验任务数。
6. 待生信分析任务数。
7. 生信分析中任务数。
8. 待交付项目数。
9. 异常项目数。

角色待办：

| 角色 | 待办示例 |
|------|----------|
| 销售负责人 | 待补充合同/委托单信息、自己负责项目状态 |
| 项目经理 | 待确认项目、异常项目、待交付项目 |
| 样本接收员 | 今日预计到样、待接收样本 |
| 实验执行员 | 待排期任务、今日实验任务、待反馈任务 |
| 生信分析员 | 待开始分析、分析中、待提交报告 |

### 6.3 项目管理

项目是平台的主业务对象。

核心字段：

| 字段 | 说明 |
|------|------|
| 项目编号 | 项目业务键，对齐**委托单编号**粒度（一委托单=一项目）；导入无来源时系统生成，规则见 §15 |
| 合同编号 | 合同标识，**可重复**（一个合同可对应多个项目/委托单） |
| 委托单编号 | 委托单标识，项目的天然业务键 |
| 客户单位 | 客户机构或公司 |
| 客户姓名 | 客户联系人 |
| 联系方式 | 电话、邮箱等 |
| 销售负责人 | 关联用户 |
| 项目经理 | 关联用户 |
| 项目类型 | 单细胞转录组、免疫组、空间转录组等 |
| 服务内容 | `service_items`，具体服务项明细（文本/多选），承接旧表「上机方式」长文本中的服务说明 |
| 服务档次 | `service_level` 枚举，**驱动流程分支**：`qc`(质控版) / `library`(建库版) / `run`(上机版) / `standard`(标准分析版) / `advanced`(高级分析版) |
| 测序平台 | `sequencing_platform`，如 Illumina / 真迈，承接旧表「上机方式」中的平台信息 |
| 项目优先级 | 普通、加急 |
| 项目状态 | 状态枚举 |
| 预计交付日期 | 项目经理维护 |
| 实际交付日期 | 交付时记录 |
| 备注 | 补充说明 |

MVP 功能：

1. 项目列表、搜索、筛选、排序、分页。
2. 新增项目（销售建草稿：不填项目编号，录入客户/合同/服务 + 样本编号 + 样本数量；项目编号由 PM 确认时录入）、编辑项目。
3. 项目详情页。
4. 项目状态动作按钮。
5. 关联样本、实验任务、生信任务、质控记录和操作时间线。
6. 项目异常标记和异常说明。

**服务档次与流程分流**：`standard`/`advanced` 含生信分析，走「待生信分析」分支；`qc`/`library`/`run` 不含生信，实验反馈后直接进「待交付」。`service_level` 在创建或确认项目时确定，历史数据可由旧表「上机方式」文本解析得到（§15）。

**交付队列（`/delivery`，顶层导航「交付」）**：面向项目经理的交付闭环工作台，是项目列表的**聚焦视图**而非独立实体——交付即项目状态流转（待交付→已完成，确认交付即关闭，§5.1 / §7.1），不建交付表。默认队列圈定「待交付」（待项目经理「确认交付」关闭），可切到「已完成」查历史。沿用全站交互模型：行内直接执行「确认交付」（一步关闭项目，行级权限与动作可用性复用项目动作规则），点编号开项目侧滑详情，深链全页。「确认交付」对项目下「待交付」生信任务的级联见 §5.1 / §8.9。

### 6.4 样本管理

样本是实验流转的核心对象。**一委托单一组样本**：一条样本记录 = 一个样本编号(YP) + 样本数量(`sample_count`)，代表一组样本（MVP 不为每个子样本建行；终局随 `task_samples` 细化）。样本编号上游给定（客户送样/预约），系统不生成。**生命周期**：销售建项目时录入 样本编号 + 样本数量，**随项目创建生成 1 条 `waiting_arrival` 样本**（物种/组织/实验类型/运输条件留空）；到样时收样员通过「登记接收」一次补全这些信息 + 记录到样 → `received`。因此**不提供独立的「登记样本」创建入口**（样本来自项目创建或预约导入）。

核心字段（物种/组织/实验类型/运输条件**可空**，收样时补全）：

| 字段 | 说明 |
|------|------|
| 样品编号 | 样本唯一标识 |
| 所属项目 | 关联项目 |
| 样本物种 | 人、小鼠、大鼠等 |
| 样本组织类型 | 肺组织、肝脏组织等 |
| 样本数量 | 当前批次样本数量 |
| 实验类型 | 组织解离、细胞核、建库、上机等 |
| 运输条件 | 4 度、冷链、干冰等 |
| 客户取样日期 | 客户实际取样日期 |
| 预计到样日期 | 预计到达实验室日期 |
| 实际接收时间 | 样本接收员登记 |
| 接收状态 | 正常、异常 |
| 异常说明 | 破损、超温、延迟、污染等 |
| 样本状态 | 状态枚举 |

MVP 功能：

1. 样本列表、搜索、筛选。
2. 编辑样本（创建走项目创建，不单设入口）。
3. 登记接收（补全样本信息 + 记录到样，一张表单）。
4. 样本异常登记。
5. 从样本创建实验任务。
6. 按项目查看样本。

经验库用于沉淀历史上机表现，为新项目提供术前参考（这类样本能期望什么产出、是否值得上机）。

**2026-06 决策：经验数据 = 业务表里的结构化实测指标，不建独立表、不做导入**（被否决方案与依据见决策记录）。数据随业务运行自然沉淀（实验完成后录入产出指标，§9.7）。参与维度：

| 维度 | 来源 |
|------|------|
| 物种 / 组织 | `samples` |
| 建库化学 / 悬液类型(细胞·细胞核) | `experiment_tasks.run_method / suspension_type` |

| 测序量 / 捕获细胞数 / 基因中位数 | `experiment_tasks`（下机定量后补录） |

数据来源（必做）：实验完成后由 `record-metrics` 动作补录下机产出指标——主录入人是生信分析员（cellranger 结果持有者），入口在生信任务详情；非生信(`run`)项目在实验任务详情录。详见 §9.4 / §9.7。

「经验库」页（`/experiences`）是统计与经验主页（非工作台——工作台保持角色队列形态）。MVP 功能：

1. 录入产出指标（`record-metrics` 动作，§9.4）。
2. **统计图表（已实现，Recharts）**：经验库页 = 趋势 + 构成 + 产出（完成项目按月趋势 + 物种/组织分布 + 捕获细胞数按物种均值），均按行级可见范围（`buildProjectScope`）统计、脏字段 Top N 原样展示。**与工作台分工**：项目当前状态各计数属"此刻"，在工作台「项目流转」轨道展示，经验库不重复（每个指标全站只出现一次）；经验库只给工作台没有的历史/分析视角。基因中位数/活率箱线等更深的产出图随数据积累再加。
3. **相似经验检索（已实现）**：经验库页按 物种 / 组织 / 建库化学 / 悬液类型（全可选、脏字段模糊匹配）检索历史「已完成且有产出」的上机记录，给出 捕获细胞数 / 基因中位数 / 活率 的中位数 + 命中明细（术前参考），按可见范围统计。后续可做成样本/实验任务详情页的上下文嵌入。
4. 经验数据**不做生产导入**；作为 **demo seed** 灌入（`prisma/seed-experience.ts`，`SEED_DEMO=0` 可关）。样例已换为对齐版：经验表与预约表同委托单号、含 `样本名`（一行一捕获），demo seed 把经验产出挂到预约表对应项目/样本（不再造独立 `DEMO-` 壳；实现待落地，详见 §9.7 / §15.6）。

## 7. 状态设计

### 7.1 项目状态

| 状态 | 代码 | 说明 |
|------|------|------|
| 项目草稿 | `draft` | 销售或项目经理创建，信息未确认 |
| 待到样 | `waiting_sample` | 项目经理已确认、等待客户样本到达（确认项目直接进此态，无独立「已确认」中间态） |
| 已到样 | `sample_received` | 至少一个样本已接收 |
| 实验中 | `lab_in_progress` | 已进入实验任务 |
| 待生信分析 | `waiting_bioinfo` | 实验完成，等待生信 |
| 生信分析中 | `bioinfo_in_progress` | 生信分析进行中 |
| 待交付 | `waiting_delivery` | 报告或结果待交付确认 |
| 已完成 | `completed` | 已交付并关闭（确认交付即终态，`deliveredAt` 记录交付日期，无独立「已交付」中间态） |
| 项目异常 | `abnormal` | 发生异常，待处理；记录 `status_before_abnormal` 以便恢复 |
| 异常终止 | `terminated` | 项目无法继续，终态 |

**项目状态聚合驱动规则**：项目中间状态不由用户直接设置，而由下列子实体动作在同一 Prisma 事务内联动改写并记入 `operation_logs`。

| 触发动作（子实体） | 项目状态转移 |
|------|------|
| 接收样本（项目下首个样本接收成功） | `waiting_sample` → `sample_received` |
| 开始实验（项目下首个实验任务 start） | `sample_received` → `lab_in_progress` |
| 提交实验反馈（项目下实验任务全部完成）+ 服务含生信 | `lab_in_progress` → `waiting_bioinfo` |
| 提交实验反馈（全部完成）+ 服务不含生信 | `lab_in_progress` → `waiting_delivery` |
| 开始分析（项目下首个生信任务 start） | `waiting_bioinfo` → `bioinfo_in_progress` |
| 提交报告（项目下生信任务全部进入待交付） | `bioinfo_in_progress` → `waiting_delivery` |
| 确认交付（PM 动作，非聚合；列于此因含生信级联） | `waiting_delivery` → `completed`（见 §5.1） |

「全部完成/全部进入」以项目下子实体集合为判定，单条子实体动作不直接把项目跳到终态。

**样本状态的子驱动**（与上表同一事务内联动，承接 §7.2 的 `lab_in_progress`/`feedback_submitted`）：开始实验时样本 `received`/`received_abnormal`/`waiting_task` → `lab_in_progress`；提交实验反馈时样本 → `feedback_submitted`。MVP 单 `sample_id` 任务下为 1:1 精确；多任务/样本（终局 `task_samples`）下为近似，不影响项目聚合判定。

**生信任务的交付级联**（与「确认交付」同一事务内联动，承接 §7.4 的 `delivered` 与 §8.9）：项目经理确认交付时，项目下处于「待交付」的生信任务统一 → `delivered` 并落 `deliveredAt` = 项目交付日期。生信任务的 `delivered` 不由独立动作设置。

## 8. 数据校验规则

> 总则：所有跨字段的**日期比较一律按自然日**（DATE 截断后比较），同日视为通过。`received_at` 等保留时分仅用于展示/排序，不参与日期先后判定。

1. **样品编号 `sample_no` 在业务样本表内唯一**（经验数据已并入实验任务，无独立经验表的唯一性例外）。
2. **项目编号唯一性**：`projects.project_no`（=委托单号粒度）在业务项目表内唯一，合同号 `contract_no` 可重复。历史导入无法保证唯一时保留原编号并生成内部 ID。
3. 样本数量、上机样本个数、测序量、捕获细胞数、基因中位数必须为非负整数（**允许为空**，未填不等于 0）。
4. 活率和结团率必须为 0-100 之间的数字（仅对结构化数值列校验；预约表埋在自由文本里的活率不在此列，见 §15）。
5. 日期字段必须为有效日期。
6. 预计到样日期不能早于客户取样日期（同日通过）。
7. 实际实验日期的自然日不能早于样本接收日期的自然日（**同日通过**）；仅当实验自然日真正早于接收自然日时，才需管理员确认异常录入。
8. 实验任务完成时必须填写结果状态（`result_status` 枚举）和结果反馈（`result_feedback` 文本）。历史导入时 `result_status` 由「结果反馈」文本启发式映射、无法判定时兜底为「其他」，原文全量保留（§15）。
9. 生信任务已交付时必须填写报告交付日期。
10. 关键状态变更必须有操作人和操作日志；历史导入动作的操作类型为 `import`、操作人为导入执行者（§6.10）。

### 15.3 导入目标状态映射（直写终态）

| 条件 | experiment_task | sample | project |
|------|------|------|------|
| 结果反馈含「正常上机」、上机个数>0 | completed | feedback_submitted | 含生信→waiting_bioinfo；不含生信→completed |

### 15.4 「上机方式」文本解析

先归一化（大小写、去换行），再派生多字段，解析失败降级为「仅存 raw_booking_text + 人工复核」，不丢数据：

- `run_method`：匹配 NEXT-GEM / GEM-X / Next GEM / 墨卓 等建库化学关键词
- `project_type`：匹配 转录组 / 免疫组（含 TCR）/ 空间转录组
- `service_level`：「质控版」→qc；「做到建库」→library；「标准分析版」→standard；「高级」→advanced；仅上机→run
- `sequencing_platform`：括号内 Illumina / 真迈
- 「做到质控/建库，不上机」→ 对应 service_level + 标记不上机

### 15.5 「结果反馈」→ result_status 启发式

优先级：含「不满足/不满足上机」→不满足要求；同时含「正常上机」与其他结论（低风险/偏黄/凋亡/不满足）→其他；纯「正常上机」→正常上机；仅含「低风险」→低风险上机；无法判定→其他。「冰袋/碎冰/到样/冷链」类识别为到样/运输信息，写 samples.transport_condition 或 abnormal_note，不进 result_status。**预约表导入不回填 qc_records**（活率等仅存于 result_feedback 文本），结构化质控数值由后续手工补录（录入质控动作）；下机产出指标（测序量/捕获细胞数/基因中位数）由 `record-metrics` 补录（§6.8），不在导入阶段处理。

### 15.6 经验表 → 不做生产导入（2026-06 决策，已更新）

经验表**不作为产品功能导入系统**；经验数据由业务运行中沉淀（§6.8 / §9.7）。早期旧 `单细胞项目经验.xlsx`（43 个 2025 委托单、101 行）另有放弃理由——数据脏（部位 肺/肺组织/肺穿刺 混写、物种混入、备注列错位）、与预约表委托单号零交集无宿主、单独 seed 需伪造 7+ 必填字段且污染运营统计——这批旧数据已弃用。

**2026-06 样例改为对齐版**：经验表换成与预约表同委托单号（`BP-G2605*`）、新增 `样本名`、每委托单行数=样本数，经验产出（一行一捕获）可 1:1 挂到预约表样本。故 demo 不再造独立 `DEMO-` 壳，而由 seed 先灌预约表项目/样本、再按 `委托单号 + 样本名` 把经验产出挂上去（`prisma/seed-experience.ts`，实现待落地）。这是 **demo seed**，非生产导入：不为一次性数据搞常驻导入 UI（KISS，历史导入已删）。如个别历史指标确需参考，由管理员通过 `record-metrics` 在对应实验任务上手工补录。
